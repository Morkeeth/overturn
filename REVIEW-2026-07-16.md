## 2. Competition read (settlement track, ~73 entries)

**Common angles in a Solana settlement/oracle hackathon** (from the track framing + how UMA/Pyth/Chainlink
resolve today):
- Render a TxLINE Merkle proof on-chain and settle a market from it. This is the obvious build and
  almost certainly what the bulk of the field ships.
- Optimistic-oracle style dispute windows (UMA pattern): propose a result, allow a challenge period.
- Off-chain-final-then-publish (Kalshi pattern: CFTC clearinghouse resolves, then pushes to Pyth/RedStone).

**Does anything already do our thing?** I pressure-tested "published != true" + schema-diff (as-of time +
supersession) against UMA, Pyth, Chainlink, RedStone:
- UMA's liveness window is a **dispute** window: it guards against a *liar*, not against the truth being
  *revised* after the fact. Re-resolution is a governance vote, not an in-schema supersession link.
- Pyth/Chainlink are append-only price streams: a correction is byte-identical to a normal move. No
  as-of vs publish distinction, no supersession pointer.
- Kalshi corrects *off-chain* before publishing, so on-chain there is nothing that expresses a revision.

So the wedge survives: **none of them express a revision in-band, at the schema level.** TxLINE does
(`Confirmed:false -> action_discarded`, batch min/max distinct from publish time).

**Strongest, most defensible framing** (and where to hold the line with Rolfe):
1. Lead with the *empirical finding*, not the abstraction: for 26 seconds a real World Cup semi-final feed
   asserted 0-3 and it was never 0-3. You only see that by reading 1,027 live events. A field that renders
   proofs did not read the stream, so they do not know it retracts.
2. Then Guard 3 (predicate binding): a genuine, valid, unforged final proof drains the escrow if the settler
   can swap the question. Found by *building*, not from docs. This is the part that reads as a real security
   result to a blockchain engineer.
3. Do NOT overclaim that the Merkle proof establishes truth. A root proves inclusion in a batch TxODDS
   published. Say exactly that. The page and README now do.

**Critical caveat to keep honest:** "the only schema that can express the difference" is strong. It is true
*in-band and on-chain*, but partly because most oracles publish already-final data. If Rolfe pushes, narrow
to the true and sharper claim: **TxLINE is a live in-play feed that emits provisional records and their
supersessions in the same channel, and that in-band revision primitive is the thing correct in-play
settlement actually needs.** In-play is where this bites; post-match settles off a final box score hours
later and nobody with money on the line gets that wrong.
