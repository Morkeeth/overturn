## 2. What is actually true (all verified today, none of it assumed)

### The finding, from real data
On 2026-07-14 (France v Spain, semi-final), TxLINE's SSE feed published Spain's **3rd goal** at
20:22:29 with `Confirmed:false`. VAR reviewed it. At 20:22:55 it was `action_discarded` and the
**score payload reverted**. `game_finalised` at 21:04:14 says Spain scored **2**.

**For 26 seconds the official feed asserted 0-3. It was never 0-3.**

The hackathon brief tells you to build exactly the thing that breaks on this:
> *"utilizes our high-speed SSE Stream ... using incoming match updates to dynamically trigger prediction resolutions on your platform"*

and marks the safe path optional:
> *"Experimental Verification Layer (**Optional**)"*

### Two vulnerabilities in the sponsor's own suggested architecture

**V1 · Settle-from-stream.** A keeper listening to SSE pays out on the phantom, irreversibly.
**V2 · Question-swap (the sharper one).** If `settle()` forwards a caller-supplied `strategy` to
`validate_stat`, **the settler chooses the question**. Our prop is "Spain > 2" (FALSE, NO wins).
An attacker holding YES submits **the same honest, valid, final proof** with the strategy changed
to "Spain > 1" — TRUE. The oracle answers honestly. The attacker takes the pot.
**No phantom needed. No forgery. Any escrow that does the obvious thing is drainable.**

I only found V2 by building the thing. It is not findable from the docs.

### What is NOT true (killed, so nobody re-derives them)
- ❌ *"The Merkle proof validates the lie"* — **false**. Proofs are built from corrected data; the
  phantom is not provable on-chain. TxODDS built the proof layer correctly. The danger is the keeper.
- ❌ *Fair-value edge vs the book* — the free tier serves **only** `TXLineStablePriceDemargined`.
  There is no second bookmaker to arb.
- ❌ *Blackout arbitrage* — measured: Polymarket repriced at 20:20:05, TxLINE's 1X2 didn't reopen
  until 20:20:50. **The venue is 45s FASTER than the feed.** The arrow points the wrong way.
- ❌ *Danger-events predict goals* — 3 and 2 danger events before the two goals vs a 2.68 base rate. Noise, n=2.
- ❌ *Fade the phantom on totals* — no totals markets exist on Polymarket/Jupiter for these matches.

## 7c. Three bugs I caught by reading output instead of trusting code

These are logged because each one would have put a false claim on screen in a video about false claims.

1. **`var` records ship a `Score` payload with no `Goals` key** → read naively, the timeline showed the
   score collapsing to **0-0** mid-VAR-review. Fixed by carrying the last asserted score forward.
2. **The fix was wrong.** `filter().map()` runs filter to completion first, so every inherited row got the
   **final** score (0-2) instead of the contemporaneous one. During the VAR review the feed's live state was
   **0-3** — the phantom still standing. Showing 0-2 there quietly rewrites the story we're telling. Fixed
   with a single pass. Rows that inherit are now labelled `inherited` on the page.
3. **`demo.ts` printed a success summary when the settlement had FAILED** — it read `yesWon` off an
   unsettled account, where the default `false` reads as "NO wins" and looks correct. A lying demo, in a
   product about lying feeds. Now asserts `state === 'settled'` and exits non-zero otherwise.

Also caught at cleanup: **`data/` was gitignored**, which would have shipped a public repo where
`npm run replay` finds nothing — and that capture is unrecoverable after ~Jul 28.
