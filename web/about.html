Overturn
Market About Story Repo
About

Everyone building on live sports data is one retraction away from paying the wrong person.

Overturn is a prop-bet escrow on Solana that refuses to settle on a result that is still provisional, still under review, or has since been withdrawn. It exists because of 26 seconds on 14 July 2026, and because the fix for those 26 seconds is not more cryptography.

What happened

France v Spain, World Cup semi-final. At 20:22:29 TxLINE's feed published Spain's third goal. It was flagged unconfirmed, which almost nobody reads. The scoreline moved to 0-3. VAR opened a review, closed it, and at 20:22:55 the goal was discarded. The score went back to 0-2 and stayed there to the final whistle at 21:04:14.

The phantom goal existed, in public, in the data, for 26 seconds.

20:21:18Goal confirmed. Spain 2. This one was real.
20:22:29Goal published, unconfirmed. The feed now says 0-3.
20:22:41VAR review opened. The 0-3 is still standing.
20:22:53VAR review closed.
20:22:55Action discarded. The third goal never happened. Back to 0-2.
21:04:14Game finalised. Spain 2. It was always 2.

A market that settled anywhere in that window paid the YES side on a goal that did not exist. It would have released the pot 41 minutes and 45 seconds before the match was over, and the chain does not have an undo button. There is no support desk. There is no chargeback.

Why a proof does not save you

TxLINE anchors every score to Solana as a Merkle proof. This is genuinely good, and it is the reason this project is possible at all: anyone can verify what the feed said and when, permissionlessly, without asking TxODDS for anything.

But run the thought all the way through. A Merkle proof of the 0-3 scoreline is a perfectly valid proof. It verifies. The signature is real, the root is anchored, the maths is honest. What it proves is that TxODDS published 0-3. It does not prove Spain scored.

A proof shows they published it, not that it happened.

the entire thesis, in one line

This is the part that surprised us, and it is why the project is not called Verify. Verifying the cryptography and settling on the result are two different jobs, and only one of them is hard. Every on-chain oracle we looked at can prove publication. TxLINE's schema is the only one we found that can also express this record was revised, because it carries an as-of time distinct from publication time and a supersession link between records. Without those two fields, a feed cannot tell you it was wrong. It can only tell you what it said.

So the escrow refuses

Settling is permissionless: anyone can call it, including a bot, including your opponent, including a stranger. That is safe precisely because the caller has no discretion. They hand over a proof; the guards and the oracle decide. Every input is checked against state frozen when the market opened, before anyone knew the answer.

All three guards run before the oracle is ever asked. The escrow refuses to verify a proof it has no business verifying, which is a cheaper and more honest failure than verifying the wrong thing correctly.

  • Guard 1, fixture binding. A real, valid, unforged proof from a different match is still a real proof. We tested this with the other semi-final's final proof: same stat key, same period, batch running past the whistle. It clears guards 2 and 3. It settles nothing here.
  • Guard 2, finality. An honest half-time proof honestly proves the half-time score, and would settle a full-match market early and wrongly.
  • Guard 3, predicate binding. Without it the settler picks the question, and picks the one they win.

Bring your own key

Nothing here is a hosted service you have to trust. The program is on mainnet and devnet at the same address, it has no admin key, and there is nobody to appeal to, including us. Everything the website does, you can do from a terminal against the same program with your own TxLINE subscription and your own wallet.

1. Your TxLINE key

TxLINE's free tier is gated by an API token you activate with an on-chain subscribe transaction (service level 12, four weeks, free). The repo has subscribe.mjs for exactly this. The guest JWT that pairs with it is minted with POST /auth/guest/start and needs no credentials. Put both in your environment and the CLI picks them up.

your key, your quota, your machine
# one on-chain subscribe, then the token is yours
node subscribe.mjs

# the CLI reads these from the macOS Keychain, or set them however you like
WC_API_TOKEN=...   # from /api/token/activate
WC_JWT=...         # from /auth/guest/start, 30 days

2. Your wallet

The maker and taker are ordinary Solana keypairs. Point RPC_URL at whichever cluster you want and the identical flow runs there. There is no special mainnet code path, which is the only reason we trust it on mainnet.

the whole product, six commands
npm run market:fixtures   # what TxLINE knows about, live
npm run market:open       # you take YES. the question freezes here
npm run market:take       # someone takes NO. the pot is now unspendable by hand
npm run market:status     # read the prop off the chain. no wallet, no signature
npm run market:settle     # hand it a proof. the oracle decides, not you
npm run market:claim      # the winner collects

# same flow, mainnet, no code change
RPC_URL=https://api.mainnet-beta.solana.com \
ORACLE=9ExbZjAapQww1vfcisDmrngPinHTEfpjYRWMunJgcKaA \
npm run market:open

3. Your agent

Every command is non-interactive, prints an explorer link, and exits with a status code, which makes the whole thing scriptable by something that is not a human. market:watch is that idea taken to its end: it follows the live feed and settles the moment the match is actually over.

It settles on game_finalised, never on a clock and never on the scoreline looking right. That is the thesis pointed back at our own tooling. An agent that trusts the feed is the thing this project exists to argue against.

What it is not

It settles one prop shape: a single stat, at a single period, against a threshold. There is no order book, no AMM, no liquidity, no fee. It is deliberately one honest primitive rather than a product surface with a broken middle.

It also cannot save you from a feed that is simply wrong and never says so. Nothing can. What it can do is refuse to act while the feed is still deciding, which is the failure that actually happened, on camera, to a real semi-final, for 26 seconds.

The escrow's most important feature is the payout it does not make.
